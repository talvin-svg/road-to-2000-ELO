## End goal (full scope — do not build ahead of the current milestone)

Import Talvin's own games from Chess.com, aggregate them into a FEN-keyed position tree to find the specific opening positions where he keeps losing to higher-rated players, then drill them by playing out the line against Stockfish (with an eval bar and AI narration) — with spaced-repetition scheduling. (The Lichess Opening Explorer was used through Milestone 4 as an interim opponent + candidate-move source; **Milestone 6 retires it** — Stockfish + AI replace it as the practice engine, so the explorer is a stepping stone, not part of the end state.)

## Roadmap

1. **Board + game replay** — ✅ done
2. **Chess.com import** — ✅ done
3. **Aggregate games into a FEN-keyed position registry, surface worst positions** — ✅ done (flat FEN-keyed map so transpositions merge — see skill inventory), see "Current status"
4. Practise problem positions against the Lichess Opening Explorer — 🚧 in progress (explorer client + interactive practice board done). Replaced with a play-the-board-get-feedback model (see Current status). Repertoire/persistence may return later, but is not the current direction.
5. Persistence + spaced repetition
6. **Stockfish + AI replay — drop Lichess entirely.** **Decision (pivot):** the Lichess Opening Explorer is removed as the practice engine. The explorer-driven play-out (weighted-random ~1000 opponent, "until out of book" seam, Masters toggle) proved not to bring enough value — its opponent only exists *inside book*, the line dies the moment it leaves the explorer, and low-band sampling gave noisy, shallow practice. Replacement: **from every problem position (the spots we lose in analysis), replay the line entirely against local Stockfish**, with an **eval bar** (a widget rendering the same local Stockfish centipawn score; NOT a separate service) and **AI (Claude) narrating the moves**. Stockfish is both the opponent (played at a chosen strength) and the ground-truth judge; blunder detection = eval swing vs the engine's best move over a threshold. No book, no explorer, no network dependency for practice — Stockfish covers the whole line from move one. **Platform strategy (decided): native only.** The `stockfish` package is FFI (native C++), so it runs on macOS/iOS/Android but **not web** (`dart:ffi` doesn't exist in the browser). Decision: **don't** take on the separate WASM-Stockfish port — instead **gate the engine on `!kIsWeb`**; on web, hide all engine UI (eval bar, blunder verdicts, engine opponent). Since practice is now engine-only, web loses the interactive practice board altogether (import/analysis still work); native is the supported target for drilling.
7. **Coaching layer (two brains: Stockfish judge + Claude coach)** — after *every* move (Talvin's call: noisy is fine, the goal is learning what to look for), Stockfish grades the move (blunder / inaccuracy / fine) and Claude narrates the engine's numbers into plain-language ideas — the plan in this structure, what to watch for, how to continue the line. **Hard rule: the LLM never decides the numbers, only explains them** (models hallucinate evals; Stockfish is ground truth, Claude turns facts into words). Graduation arc: coach heavily while a position is unfamiliar → once Talvin understands it, drop the words and just play it out against the engine. Uses the latest Claude model; API key injected via `--dart-define-from-file` exactly like `LICHESS_TOKEN`.

Only build the current milestone. Don't scaffold later milestones early "for convenience" — the whole point is small, discussable steps.

## Current status

Milestone 3 complete. Import games from Chess.com → app accumulates games into a FEN-keyed registry → "Problem Positions" screen shows boards ranked by loss rate, split by White/Black. Milestone 4 in progress. The Lichess Opening Explorer is wired up and drives a **play-the-line-out board** (`features/analysis/position_detail_screen.dart`): open a problem position and **play the line out** — you play the move you'd actually play, it's graded against the explorer (verdict names the main move if you missed it), then **the app replies as a ~1000 opponent** and you continue until the line **leaves book** (no games at this level) or the game ends. A running move list (your moves gold, opponent's muted) shows the line so far. A "Your level / Masters" toggle switches the database (and restarts the line). Nothing is saved — pure practice.

**Opponent model (this milestone's core decision):** the ~1000 opponent is **not** an engine — it's a **weighted-random pick** from the `/lichess` explorer's low rating band for the position after your move (weighted by how often each move is actually played there). Rationale: for openings, the low-band explorer *is* a sample of real 1000-rated human moves — including their habitual mistakes — so it's a *more* faithful 1000 than a throttled Stockfish (which plays unnatural engine moves then random blunders). A weakened engine is both the wrong tool here and Milestone 6 work. The **"until out of book" stop condition is the seam** where the engine will later take over (book → explorer, beyond book → engine).

**Play-out vs the old snap-back (design change):** the board used to *snap back* after every move so top-move arrows could overlay the original position and you could experiment freely. Play-out **applies** your move and advances, so those arrows can't overlay a position that's no longer on the board — the feedback became **textual** (verdict names the main move) plus the move list. Talvin chose to convert the practice board rather than keep both modes, so the snap-back experiment mode is gone.

**Design pivot worth remembering:** this replaced an earlier repertoire + drill design (tap a candidate off a text list to "commit" it, then a separate drill screen quizzed you on it). Talvin rejected it — being asked to pick a move he *couldn't see* off an abstract SAN list was a bad learning UX. Lesson: for a learner, board-first ("play it and I'll tell you") beats list-first ("choose the right answer from names"). The repertoire model, `RepertoireController`, and `DrillScreen` were deleted.

**Flow restructured** so the two purposes aren't tangled on the month row: tapping a month → browse its games → pick one → **replay** (the single-game path); **Add** → month joins the analysis pool; a **pinned "Analyze N games" bar** (shown once the pool is non-empty) opens Problem Positions → tap a position → the practice board. Analysis is no longer buried behind the replay screen. In replay, the old "Import game" button became **"Pick another game"**, which returns to the month list (`backToMonths()` then push).

**Best-move arrows are back (as a miss flash).** The arrow feedback lost in the snap-back → play-out conversion has returned, reshaped for play-out: when you *miss* the top explorer move, the board holds the pre-move position for ~1s and draws two arrows — **your move (warm red) vs the top move (green)** — then applies your move and the opponent replies. Play the top move on the nose and nothing flashes (quiet reinforcement, no nagging). The flash sits *between* grading and `_applyMove` precisely because the best move belongs to the position you moved in — once the line advances, that position is gone, so the arrows have to be shown before it is. Board is locked during the flash (`_hintLocked`).

Next: options from here — a session that cycles through several problem positions; an always-on "show me first" hint before you move (distinct from the miss flash — teach vs test); or persistence + spaced repetition (Milestone 5). Milestone 6 (Stockfish + AI) now **replaces** this explorer-driven practice entirely (see roadmap): Lichess is dropped and the whole line is replayed against local Stockfish with an eval bar and AI narration — so the "until out of book" seam and the explorer opponent are going away, not being handed off to.

## Environment gotchas — don't "fix" these away without understanding why they're there

- **Flutter is pinned to 3.32.0**, globally on this machine (not project-local) — macOS 13.0 can't run Flutter's tool binary on 3.4x+ (needs macOS 14+). If Flutter seems "out of date," that's why — upgrading will break the tool entirely on this OS.
- **`--wasm` is required** to run this app on web at all. `dartchess`'s `SquareSet` bitboard code uses raw 64-bit integer literals (e.g. `0xffffffffffffffff`) that the standard JS compiler (dart2js/dartdevc) rejects outright — only Dart's WASM compile target has real 64-bit integers. This isn't optional polish, the app does not compile for web without it.
- `pubspec.yaml` has a `dependency_overrides: meta: ^1.18.0` — chessground 10.x needs a newer `meta` package than Flutter 3.32.0's bundled `flutter_test` provides. Verified this override is safe by checking meta's changelog (the 1.17 → 1.18 diff is purely additive annotations, nothing removed).

### What I've actually learned so far (running log — append short entries as real understanding lands, not just exposure)
- **`core/` vs `features/` split** — `core/` is UI-agnostic chess logic, `features/` builds on top of it. Dependency flows one way: features → core, never the reverse.
- **PlyRecord and GameReplay** — why a flat indexed list of plies beats replaying moves on every jump; why FEN strings are stored instead of `Position` objects (cheaper, sufficient for the UI).
- **`Position` immutability** — `.play()` returns a new object rather than mutating in place; the loop discards old `Position`s because `PlyRecord` already captured everything the UI needs.
- **Riverpod basics** — `ref.read` (once, no subscription), `ref.watch` (rebuild on change), `ref.listen` (side effect on change without rebuild). `NotifierProvider` exposes state via `ref.watch` and the notifier itself via `.notifier`.
- **`copyWith` pattern** — produces a new state object with one field changed; required for Riverpod to detect the change and trigger a rebuild.
- **Sealed class state machines** — `sealed class ImportState` with subclasses for each step of the import flow. The UI `switch`es exhaustively on the state so every case is handled and the compiler catches missing ones.
- **Switch expressions with destructuring** — `switch (state) { SelectingMonth(:final List<String> archives) => archives, _ => [] }` pulls a field out of a subtype and produces a value inline. The `:fieldName` syntax is shorthand for "match this subtype and bind its field".
- **`PopScope`** — intercepts back navigation in Flutter. `canPop: false` blocks the default pop; `onPopInvokedWithResult` runs instead, letting you transition state (e.g. game list → month list) rather than leaving the screen entirely.
- **`ref.listen` for side effects** — when Riverpod state changes need to update something outside the state tree (like a `TextEditingController`), `ref.listen` runs a callback without triggering a rebuild.
- **`??` null coalescing** — `expr ?? fallback` returns `fallback` when `expr` is null. Used when reading optional PGN headers that may not exist in every game file.
- **Platform entitlements and permissions** — macOS sandboxes Flutter apps; outbound HTTP requires `com.apple.security.network.client` in the entitlements plist. Android requires an explicit `INTERNET` permission in `AndroidManifest.xml`. iOS allows HTTPS by default.
- **REST API consumption** — `http.get(Uri.parse(...))` for a GET request; check `response.statusCode` before reading `response.body`; `json.decode` to parse the JSON into a Dart map.
- **State-driven routing without a router package** — `AppRouter` is a plain `ConsumerWidget` that watches a Riverpod state and returns a different widget tree based on it. No `Navigator.push`, no `go_router` — the framework reconciles the widget tree automatically. Used here so `ImportScreen` is the root until a game is loaded, then `ReplayScreen` takes over.
- **`shared_preferences` for lightweight persistence** — key/value storage backed by platform APIs (UserDefaults on iOS/macOS, SharedPreferences on Android). `SharedPreferences.getInstance()` is async but cached after the first call. Load on Notifier `build()` via fire-and-forget async, save/clear alongside state mutations.
- **Fire-and-forget async in Notifier** — calling an `async` method from a synchronous `build()` without `await` is valid when you want the state to update immediately and the async work to follow. The synchronous state change triggers a rebuild right away; the `await` continuation runs later on the event loop.
- **Position registry (FEN-keyed, flat)** — every unique position is one node storing its full FEN, its ply depth, and a W/L/D scoreboard. Built by walking each imported game move by move; at each position where it's Talvin's turn, the outcome is recorded on the node *before* the move (the position he had to find a move in). Positions live in a flat `Map<String, PositionNode>` keyed by a **normalized FEN** (first 4 fields — pieces, side-to-move, castling, en passant — dropping the two move counters). This is what merges **transpositions**: the same position reached via different move orders is one node with pooled stats.
- **Why flat, not a tree (key lesson)** — the earlier version was a *tree* keyed by the move played, which counted transpositions as separate nodes. Switching the key to FEN alone would NOT have fixed it: in a tree you reach a node by descending from a parent, and transpositions have different parents, so they'd still split. Merging transpositions *requires* abandoning the tree for a flat FEN→node map. Naming matters: the class is still called `PositionTree` but is really a registry.
- **`worstPositions` is now filter-and-sort** — with a flat map there's no recursion: filter `_positions.values` on depth ≥ `_minPly`, total ≥ `minGames`, and matching colour-to-move; sort by loss rate descending; take top N. (The old recursive `_collectNodes` depth-first walk was only needed because the data was a tree.)
- **Derived `Provider`** — a `Provider<T>((Ref ref) { ... })` is a computed value that depends on other providers via `ref.watch(...)`. It re-evaluates only when its dependencies change, and caches the result between rebuilds. Used here to build the position tree once per import batch and share the result with any screen that needs it.
- **`cross-provider` reads in Notifier** — inside a `Notifier` method (not `build()`), `ref.read(otherProvider.notifier)` accesses another notifier to call its methods. Used in `ImportController` to push games into `GamesController` and to clear the replay state when the user resets.
- **`FutureProvider` + `AsyncValue`** — a provider that wraps an `async` function and exposes an `AsyncValue<T>` (the three states of a fetch: loading / data / error), unwrapped in the UI with `.when(loading:, error:, data:)`. Failure is signalled by *throwing* inside the provider (becomes `AsyncValue.error`), so we translate our `Result` `Failure` into a thrown `Exception`.
- **`FutureProvider.family`** — turns a provider into a function of an argument: `explorerProvider(query)` returns a provider dedicated to that argument, each cached independently. The argument is the cache key, so it needs value-equality — hence a **record** (`({String fen, ExplorerSource source})`) as the key. In Riverpod 3 providers auto-dispose by default, so leaving the screen frees the query's cache. Type `FutureProviderFamily<T, Arg>` is exported from `flutter_riverpod/misc.dart`, not the main barrel.
- **Extracting a shared rule to one source of truth** — the FEN normalization was private to `PositionTree`; it got pulled out to a top-level `normalizeFen(String)` in `core/chess/fen.dart` when a second feature needed to key positions identically (two byte-identical copies of a rule is a bug waiting to happen). That second feature was later removed, so it's a one-caller helper again — but the lesson holds: the trigger for extracting is "a second caller now needs the *exact* same behaviour," not "it might be reused someday."
- **`ref.read` vs `ref.watch`** — `ref.watch` in a `build` method rebuilds the widget when the value changes (used to redraw from explorer/analysis state); `ref.read` grabs a value once without subscribing, for one-shot reads inside callbacks. Reaching for `watch` in a callback (or `read` in `build`) is the usual Riverpod mistake.
- **Interactive chessground board** — a view-only board is `GameData(playerSide: PlayerSide.none)`; to accept moves, set `playerSide` to the side to move and supply `validMoves` via dartchess `makeLegalMoves(position)` (a `Map<Square, Set<Square>>`). The `Chessboard` widget's `onMove(Move, {viaDragAndDrop})` callback fires when the user completes a move. The controller does NOT auto-apply it — you decide: call `controller.updatePosition(newGameData)` to accept, or leave the controller unchanged and the dropped piece snaps back to its origin. The practice board exploits the snap-back: after *every* attempt it resets to the starting position so the top-move arrows overlay the original spot and the next try starts clean.
- **Drawing on the board (annotations)** — the `Chessboard` widget takes a `shapes: Set<Shape>` param; `Arrow(color, orig, dest)` and `Circle(color, orig)` (from chessground) draw an arrow between squares and a ring on a square. Use semi-transparent colours so the pieces stay visible. To turn an explorer move's UCI into squares, `NormalMove.fromUci('g8f6')` gives `.from`/`.to` as `Square`s — exactly what `Arrow`/`Circle` want.
- **Parsing a FEN into a live position** — `Chess.fromSetup(Setup.parseFen(fen))` turns a FEN string into a dartchess `Position` you can query (`.turn`, `.legalMoves`) and advance (`.play(move)` → new `Position`). It's the source of truth for whose move it is and what's legal; the board is just a view of it.
- **Matching a played move by UCI** — `Move.uci` gives a stable string ('e2e4', 'e7e8q' with promotion), so finding the user's move in the explorer list is `moves.indexWhere((m) => m.uci == move.uci)` (-1 = not in the database). Known caveat: castling has two UCI encodings (king-two-squares `e1g1` vs king-takes-rook `e1h1`); if that ever bites, compare the *resulting position's FEN* instead of the UCI string.
- **Ephemeral vs durable state (where state belongs)** — the practice board's transient state (the move you just tried, whether arrows are shown) lives in the screen's `setState`, NOT a global Riverpod Notifier, because it's meaningful only while that screen is open. Durable/shared data (the imported games, the explorer cache) lives in providers. Rule of thumb: reach for a Notifier when state must outlive the widget or be shared; keep genuinely screen-local state local.
- **`.future` on a FutureProvider — imperative await vs reactive watch** — `ref.watch(explorerProvider(q))` gives an `AsyncValue` that rebuilds the widget when the fetch resolves (reactive, for `build`). `ref.read(explorerProvider(q).future)` gives the raw `Future<T>` you can `await` inside a callback (imperative, one-shot). The play-out loop needs the second form: after you move, the opponent must fetch the explorer for the *new* position mid-callback and act on it — not rebuild the UI. Same provider, so the two share one cached fetch (watch and read don't double-hit the network). Always `if (!mounted) return;` after an `await` in a `State` before calling `setState`, since the screen may have been popped meanwhile.
- **Advancing a live `Position` + `makeSan`** — the board went from a fixed `late final Position` (snap-back) to a **mutable** one that advances as the line is played. `position.makeSan(move)` returns a record `(Position, String)` — the new position *and* the SAN in one call (the SAN must be computed before the move is applied, so doing both together is why the API is shaped that way). Whose turn it is (`_position.turn`) and game-end (`isGameOver` / `isCheckmate` / `isStalemate`) are read straight off the position — no separate bookkeeping.
- **A turn loop as an imperative async method** — `_onUserMove` → grade → `_applyMove(yours)` → `_opponentReply()` (await explorer → weighted-pick → `_applyMove(theirs)` → check end). Keeping termination *entirely* in this linear flow (rather than half in `build`) is what makes it readable: every "line over" path (`isGameOver`, empty explorer for either side, load failure) funnels through one `_endLine(reason)`. The board is locked between your move and the reply via a `_opponentThinking` flag that `_gameForCurrent()` reads to set `PlayerSide.none`.
- **Weighted-random sampling** — pick an item proportional to a weight: sum the weights, roll `Random().nextInt(total)`, walk the list subtracting each weight until the roll goes negative — that item is the pick. Used to make the opponent play common moves often and rare ones occasionally (a realistic 1000), instead of always the single most-popular move.
- **Grade against a captured snapshot, not live state** — the verdict on your move must be computed against the explorer result for the position you *played it in*. Once the line advances, `build`'s watched result is for a *different* position, so the result is snapshotted into `_gradedResult` at move time and the verdict renders from that — a reminder that reactive state (`watch`) reflects *now*, so anything about a past moment has to be captured when it happens.
- **A design consequence of an interaction change** — converting snap-back → play-out didn't just move code; it *removed an affordance*: the best-move arrows overlaid the starting position, which play-out no longer keeps on screen, so that feedback had to become textual. Worth internalizing: changing *how* the user interacts can invalidate a whole feedback mechanism, not just its plumbing.
- **Re-earning a lost affordance by inserting a beat, not reverting the model** — the best-move arrows came back without undoing play-out. The trick was *when*: an `await`ed pause is slipped **between grading your move and applying it**, so the board still shows the position the arrows belong to. A visual annotation is only truthful over the position it describes — so if the position is about to change, the annotation has to be shown *before* the change, not drawn on the aftermath. (Drawing the arrow on the already-advanced board would point its origin at an empty or wrong square.)
- **External `shapes` vs user-drawn shapes** — chessground's `Chessboard` takes a `shapes: Set<Shape>` param that is *separate* from the shapes a user draws by dragging (those live in the controller). App-supplied annotations (hints, engine arrows) go through `shapes:`; `Arrow`/`Circle`/`Shape` are exported from the barrel. Semi-transparent ARGB colours (e.g. `Color(0xCC…)`) keep the pieces legible under the arrow.
- **Locking a board that's still on your turn** — during the flash the position hasn't advanced, so `_position.turn` is still yours and the board would happily accept a *second* move over the arrows. A dedicated `_hintLocked` flag feeds the same `interactive`/`canMove` gates as `_opponentThinking` (→ `PlayerSide.none`) to freeze it. Lesson: "whose turn is it" isn't always enough to decide if the board should accept input — transient UI states need their own lock.
- **Environment variables — runtime vs compile-time (key distinction)** — an env var is a `KEY=value` entry in a process's environment dictionary, inherited from the parent process (the shell) and living only as long as that process. Normal runtime lookup in Dart is `Platform.environment['X']`. But `String.fromEnvironment('X')` is **different**: it's a *compile-time* constant baked in by `--dart-define=X=...` at build time, not read from the OS at runtime. Dart does this because Flutter's web/AOT targets have no OS process environment to read from — so the value is resolved when a shell definitely exists (build time) and hardcoded into the binary. `--dart-define-from-file=file.json` is the same mechanism fed from a JSON file. Because the source only ever contains the *name*, secrets stay out of git; the value flows gitignored file → build flag → binary → `fromEnvironment`. Used for `LICHESS_TOKEN`. (Enterprise scales this up: secret managers/Vault the running service reads via its own machine identity, per-env dev-vs-prod secrets, CI secret stores, short-lived credentials — but the core rule is the same: keep the value out of code, feed it from the environment.)
